snaPDF — CI / CD Pipeline

GitHub Actions → ECR → GitOps → ArgoCD → EKS

Automated
Manual approval (prod only)
Not yet wired
👩‍💻
Developer
Writes code in api/, worker/, or auth/ and pushes to the snaPDF repo.
main→dev
staging→staging
prod→prod
🐙
GitHub — snaPDF repo
Push triggers the path-scoped workflow for the changed service only. Pushing to api/ never rebuilds the worker image.
ci-api.yml ci-worker.yml ci-auth.yml
⚙️
GitHub Actions — CI Pipeline
Every workflow runs the same sequence. Stops at any failing step.
① Checkout ② AWS OIDC auth ③ ECR login ④ Lint (flake8) ⑤ Unit tests (pytest) ⑥ Docker build + push ⑦ Update gitops repo
📦
ECR — Image Registry
Image pushed with two tags: the git SHA for traceability and :latest as an alias.
snapdf-api : <sha> snapdf-worker : <sha> snapdf-auth : <sha>
📝
GitOps Handoff — snaPDF-gitops repo
CI clones the gitops repo, updates the image tag in the correct environment values file, and pushes. CI never touches the cluster directly.
environments/dev/api/values.yaml image.tag: abc1234
🔄
ArgoCD — GitOps CD
Detects the git change via webhook (instant) and runs helm upgrade with the new values.
dev → auto-sync staging → auto-sync prod → manual approval
⚠️ Prod requires click-to-sync in ArgoCD UI
☸️
EKS — New pods roll out
Rolling deployment — old pods stay alive until new pods pass health checks. Zero downtime.
dev namespace staging namespace prod namespace
